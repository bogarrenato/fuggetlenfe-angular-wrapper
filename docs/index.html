Angular Wrapper

Standalone Angular integration for the shared primitive layer.

The Angular wrapper keeps product teams inside a familiar Angular template model while the shared Web Component core continues to own behavior, accessibility, and public runtime semantics.

Getting started

Angular consumers import the standalone wrapper exports into their component and pull the shared token contract plus the selected brand stylesheets into the application styles.

1. Install packages
pnpm add @fuggetlenfe/angular-wrapper @fuggetlenfe/tokens @fuggetlenfe/brand-styles
2. Import the standalone directive
import { Component } from '@angular/core'; import { FfButton } from '@fuggetlenfe/angular-wrapper'; @Component({ selector: 'app-root', imports: [FfButton], templateUrl: './app.html' }) export class AppComponent {}
3. Add token stylesheets to your global styles
@import '@fuggetlenfe/tokens/contract.css'; @import '@fuggetlenfe/brand-styles/brand-1-light.css'; @import '@fuggetlenfe/brand-styles/brand-1-dark.css';

Architecture boundaries

The Angular wrapper exists so Angular teams can consume the shared primitive layer through standalone imports and template bindings. It should not grow into a second design system or a place for application-specific decisions.

What belongs here

Standalone Angular exports generated from the shared Stencil core.
Property and event bridge behavior for Angular templates.
Typed integration surface for Angular product teams.

What does not belong here

Theme activation or brand ownership.
Business workflows or page-level layout rules.
Any hardcoded styling that should live in the brand library.

Component reference

The Angular wrapper preserves the same public primitive API that exists in the Stencil layer. The PoC is intentionally limited to the button because that is the primitive currently represented in Figma and covered by the sync contract.

Primitive Standalone import

FfButton

Shared action primitive for shell actions, content actions, and theme demonstrations. The Angular wrapper exposes it in a template-friendly form without injecting design logic.

<main data-brand="brand-1" [attr.data-theme]="activeTheme()"> <ff-button type="submit">Save changes</ff-button> </main>
PropertyTypeDefaultDescription
label string undefined Fallback label when no projected content is provided.
disabled boolean false Disables pointer and keyboard interaction. Forwarded to the native button.
fullWidth boolean false Expands the host to 100% inline size without coupling layout to a specific brand.
type 'button' | 'submit' | 'reset' 'button' Mirrors the native button type attribute. Use 'submit' inside forms.

Every visual characteristic of ff-button is driven by CSS custom properties defined in the shared token contract. Brand packs populate these tokens. Consumer applications may override them at any specificity level.

TokenDescriptionDefault
--ff-button-bg-default Background color in the default (rest) state. From brand pack
--ff-button-fg-default Foreground (text/icon) color in the default state. From brand pack
--ff-button-bg-hover Background color on hover. From brand pack
--ff-button-fg-hover Foreground color on hover. From brand pack
--ff-button-bg-active Background color on press (active). From brand pack
--ff-button-fg-active Foreground color on press (active). From brand pack
--ff-button-bg-disabled Background color when the button is disabled. From brand pack
--ff-button-fg-disabled Foreground color when the button is disabled. From brand pack
--ff-button-radius Border radius of the button surface. From brand pack
--ff-button-padding-inline Horizontal (inline-axis) padding. From brand pack
--ff-button-padding-block Vertical (block-axis) padding. From brand pack

CSS shadow parts

The component exposes named shadow parts so that consumer stylesheets can target internal elements through the ::part() pseudo-element selector without relying on implementation details.

PartDescription
button The native <button> element inside the shadow DOM. Accepts any CSS property including font, padding, border, and transition overrides.
Targeting a shadow part
ff-button::part(button) { transition: background-color 180ms ease, color 180ms ease; }

Slots

Content projection uses the standard Web Component slot mechanism. The Angular template projects child content into the default slot automatically.

SlotDescription
(default) Button content. Any inline or block-level HTML is accepted. Falls back to the label property value if the slot is empty.
Projected content example
<ff-button> <svg aria-hidden="true" width="16" height="16"><!-- icon --></svg> Save changes </ff-button>
Fallback label example
<ff-button label="Save changes"></ff-button>

Theming and brand activation

Angular applications activate styling by importing the token contract and the official brand pack, then setting the visual context on the outer shell with data-brand and data-theme.

Brand and theme context on the shell
<main class="consumer-shell" data-brand="brand-1" [attr.data-theme]="activeTheme()"> <ff-button>Launch selected brand</ff-button> </main>
Runtime theme switching with a signal
import { Component, signal } from '@angular/core'; import { FfButton } from '@fuggetlenfe/angular-wrapper'; type Theme = 'light' | 'dark'; @Component({ selector: 'app-shell', imports: [FfButton], templateUrl: './shell.html' }) export class ShellComponent { readonly activeTheme = signal<Theme>('light'); toggleTheme(): void { this.activeTheme.update( (current: Theme): Theme => current === 'light' ? 'dark' : 'light' ); } }

Supported override paths

Application shell data-brand and data-theme context.
Documented CSS custom properties from the shared token contract.
Host classes passed from the Angular template.
Shadow part selectors for deep structural overrides.

Token architecture

The design system uses a three-layer token model that separates the design source of truth (Figma) from the runtime contract consumed by components. Understanding this pipeline is essential for teams who maintain brand packs or create custom themes.

The three-layer model

Layer 1 — contract.css
Declares every CSS custom property the component layer can reference. This file contains no values, only empty variable declarations. It acts as the schema.
Layer 2 — figma-preset.css
Generated by the Figma sync pipeline. Provides a default set of resolved token values that match the current Figma component definitions exactly.
Layer 3 — Brand pack
A scoped stylesheet keyed to [data-brand] and [data-theme] selectors. Each brand pack overrides the contract variables for a specific visual identity.
Token resolution pipeline
Source Sync Contract Brand Component Figma ──> figma-sync ──> contract.css ──> brand pack ──> ff-button variables CLI tool empty decls values by reads vars & styles extracts define the brand+theme at runtime tokens public API selector
How data-brand and data-theme activate tokens
/* contract.css - Layer 1: declares the schema */ :root { --ff-button-bg-default: ; --ff-button-fg-default: ; --ff-button-radius: ; } /* brand-1-light.css - Layer 3: populates values */ [data-brand="brand-1"][data-theme="light"] { --ff-button-bg-default: #de6b24; --ff-button-fg-default: #ffffff; --ff-button-radius: 12px; } /* brand-1-dark.css - Layer 3: different values, same contract */ [data-brand="brand-1"][data-theme="dark"] { --ff-button-bg-default: #f4a261; --ff-button-fg-default: #1a1210; --ff-button-radius: 12px; }

Creating a custom brand pack

A custom brand pack is a CSS file that targets a unique data-brand value and populates every token declared in contract.css. The component layer does not need to change. Import the custom pack instead of the official one.

Copy contract.css as your starting template.
Replace each empty declaration with your brand values.
Scope values under [data-brand="your-brand"] and [data-theme] selectors.
Import the file at the application shell level alongside the contract.

Accessibility expectations

Accessibility behavior belongs to the shared primitive layer, but the Angular wrapper must preserve that contract through template bindings and event handling.

AreaCurrent expectation
Keyboard supportButtons keep native button semantics. Dropdown supports Arrow keys, Enter, Escape, and Tab.
ARIA modelDropdown exposes a listbox pattern with proper selected and expanded state through the primitive.
Theme qualityOfficial brand packs are expected to meet WCAG AA contrast targets before release.
Focus indicatorUses :focus-visible so focus rings appear only for keyboard navigation, not pointer clicks.
Color contrastAll foreground/background token pairs in official brand packs must maintain a minimum 4.5:1 contrast ratio for normal text.

Keyboard navigation

The button primitive uses a native <button> element, which provides built-in keyboard behavior. The following table documents the expected interactions.

KeyBehavior
Enter Activates the button. If the button has type="submit", submits the enclosing form.
Space Activates the button. The activation fires on key-up, matching native browser behavior.
Tab Moves focus to the next focusable element in the document tab order.
Shift+Tab Moves focus to the previous focusable element in the document tab order.

ARIA

Because the component uses a native <button> element inside its shadow DOM, most ARIA semantics are provided implicitly by the browser. Consumers should not need to add ARIA attributes for standard button usage.

Implicit behavior

The native <button> element carries an implicit role="button". Do not add a redundant role attribute.
The disabled property is forwarded to the native disabled attribute, which browsers expose as aria-disabled="true" automatically.
Focus indicators use :focus-visible to show a visible ring for keyboard navigation while suppressing it for pointer interaction.
Projected text content serves as the accessible name. If only an icon is projected, add an aria-label attribute to the host element.
Icon-only button with accessible label
<ff-button aria-label="Close dialog"> <svg aria-hidden="true" width="16" height="16"> <path d="M4 4l8 8M12 4l-8 8" stroke="currentColor" stroke-width="2" /> </svg> </ff-button>

Release and maintenance model

The wrapper is intended to ship as part of a centrally versioned platform. Product teams should consume it as a package. Shadcn-style copy ownership may exist only as an escape hatch for exceptional product requirements, not as the default path.

Recommended enterprise policy

Default: consume the official wrapper package and official brand styles.
Exception: copy ownership only with explicit ownership and upgrade responsibility.
Never fork the official corporate brand pack as the default delivery model.
